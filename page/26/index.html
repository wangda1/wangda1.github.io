UNLINK

unlink 中的很重要的一点就是合并

  • 常用的两道检查（这里以后向合并为例，一般构造为后向合并）
  1. 判断两个块的 size 是否正常（即：size 未经修改）

（这里的P是指 需要 unlink的 P，也就是 目前在 bins中的 P）

  • 下面图来看
  • 当我们时，若进行后向合并（合并低地址）则 进行的操作为：
  1. BK = P -> bk
  2. FD = P ->fd
  3. FD -> bk = BK （ptr - 0x10(64bits)）
  4. BK -> fd = FD (ptr - 0x18(64bits))
  • unlink之后的ptr 的内容修改为 ptr - 0x18!!!!至此任务完成



小结：完成unlink需要：先free出两个 chunk ，然后再把 构造 chunk 内容，然后 free 中间指针完成 unlink

return i;

1. 为什么在leak atoi@got时，要多发送 0x18个a


答：因为在unlink之后 ptr[0] 保存的是 ptr -0x18，这时候edit转到 ptr 所在地址块，通过 ‘a’ * 0x18 + atoi@GOT巧妙地把它填入我们可以读出内容的地址 ptr

2.为什么修改完地址后，覆盖成 system的地址可以直接执行


答：因为此时 ptr 保存的为 atoi@GOT，再次edit ptr[0]，即编写 GOT表的内容因此此时修改的地址转到 GOT表，我们将其修改为 system@GOT，下次调用 atoi 实际调用 system （write-anywhere-anything是很可怕的）

详细摘录一下heap内存的分配策略

应当好好阅读 glic 中关于 ptmalloc 的源码

  • 根据用户 malloc 的 size 计算实际要分配 size

  • 当 size < max_fast（默认是64B）在 fast bins中取

  • 当 size > max_fast 或 在fast bins中没找到，在 small bins （< 512B）找

  • 当 不满足 small bins的条件（fast bins和small bins均为精确查找），开始合并 fast bins（相邻的合并）连接到 unsorted bins，查找，若不满足，则将 unsorted bins归类到 small bins或 large bins

  • （注意，此时 all bins only small bins and large bins left）

  • 在 large bins中“smallest-first,best-fit”寻找分配，剩余link to the bins

  • above is not fit.OK, go to the top chunk

  • Top Chunk也不满足时，开始大招–（你到底是不是第一次调用？为什么都满足不了你）
  • 若你确实需要很多 >= mmap()阈值—syscall：1. main arena :调用 sbrk()增加 top chunk；2. thread arena：调用 mmap()增加一个 sub-heap
  • 若你是第一次调用 =>进行一系列初始化工作

关于 ASLR 机制

发表于 2018-05-08 | 分类于 CTF ， PWN

关于ASLR机制

ASLR(Address Space Layout Randomization)地址空间布局随机化
分为3级：0,1,2

  • 0：没有随机化。
  • 1：保留的随机化。共享库，栈，mmap( )及VDSO将被随机化
  • 2：完全的随机化。在1的基础上，通过brk( )分配的内存也将随机化

UAF & Double Free

  • UAF( use after free)

释放重用漏洞，具体导致原因是对释放的指针没有赋值为空，



  • Double Free
  • https://bbs.pediy.com/thread-225688.htm
  • 通过这篇文章，好像有点明白Double Free的意思了：臆测一下Free的过程：
  1. 检测是否为NULL，若为NULL则报错？否则进行释放操作
  2. 检测是否满足合并条件（前向或者后向合并），若满足进行 unlink 操作

  2. 利用 fastbin + 堆溢出 修改邻近 chunk 标志位，释放，达到 unlink

heap学习篇（二）

前记：
还记得，上次关于 Chunk图， fd ,bk指针没说他们的作用，这里就用到它们了，用于显示链表哦（什么是显示，隐式链表呢？显示链表真链表有指针互指哦），显示链表在这里的引出（不要忘了，它只有在 Free Chunk中存在），肯定是为了 Chunk 分配的效率提升考虑啊


  • bin 是记录 Free Chunk 的双链表数据结构
  • 还记得吗？它是定义在 malloc_state(Arena Header)的数组哦
  • bins包含2种：Fast Bins 和 Bins


用图显示：


  • 关于 Fast Bin
  • Chunk Size: 16–80Bytes
  • Fast Bin是个单链表（只使用 fd指针），因为只对链表尾操作，后进先出(LIFO)
  • 10个 Fast Bin按大小排列 16Bytes–24Bytes–80Bytes
  • 不会对Free Chunk进行合并操作（因为Fast Bin 就是为了快速小内存分配），因此P标志位是1（表示假分配）
  • malloc(Fast Chunk)操作顺序：当第一次操作（Fast Bins为空）— small bin处理（若为空）— malloc_consolidate函数对 malloc_state初始化



  • 关于Unsorted Bin
  • 引入这个的目的是第二次机会利用最近释放的 Chunk（第一次机会就是 Fast Bin机制了），加快内存的分配与释放
  • 对Chunk size 没有限制
  • 是个循环双向链表



  • 关于Small Bin
  • Chunk Size 小于 512Bytes
  • 两个相邻Small Bins大小相差8Bytes
  • Small Bins个数：62个
  • 是个循环双向链表，可进行合并操作


  • 关于Large Bin
  • Chunk Size 大于512Bytes
  • Large Bins数量：63个
  • 是个循环双向链表，可进行合并操作
